A good navigation stack is one that is **versatile and robust**. For instance, oftentimes, it isn't viable to test the physical robot; however, without testing, it is generally unreasonable to make meaningful changes without any bugs. As such, it is best to design your code to be **adaptable, future-proof, and self-documenting** to function in as many environments as possible. The physical robot, Gazebo, and even simulation environments such as Unity are all tools that may be used to determine if the navigation code is working as intended. This is why we use **modular plugin systems** for our simulation, navigation algorithms, sensors, and state logic, so our code can function everywhere.

The standard topics/info that should be handled are
- `/tf` for a full TF tree with frames (`/map` -> `/odom` -> `/base_link`)
- `/odom` helps to generate pose through the TF tree
- `/scan` or `/point_cloud` for costmaps
- `/cmd_vel` from Nav2 and multiplixed by twist_mux
- `/joint_states` for control and awareness of sensor positions
- `robot_description` for `robot_sate_publisher` to create the appropriate transform tree

To minimize strange hidden aspects of the code, avoiding (or better documenting) MAVROS and sticking to in-house hardware level interfaces for `ros2_control` with our own physics engine for omnidirectional (or the one integrated into `ros2_control` if it works for our use case) may make more sense.
